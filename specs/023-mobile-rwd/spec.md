# 功能規格：行動裝置響應式佈局（Tab 切換模式）

**功能分支**: `023-mobile-rwd`
**建立日期**: 2026-03-11
**狀態**: 草稿
**輸入**: 使用者描述：「為 Semorphe UI 實作響應式設計（RWD），在行動裝置上使用 tab 切換模式。目前佈局為僅限桌面的左右 50/50 分割面板（Blockly 積木面板 + Monaco 程式碼編輯器）。在行動裝置（≤768px）上，以全螢幕 tab 切換 UI 取代分割面板：底部 tab bar 含 3 個分頁（積木/程式碼/主控台），每個面板佔滿整個螢幕寬度。桌面佈局（≥769px）維持不變。在行動裝置上簡化工具列（次要功能收入漢堡選單）。隱藏狀態列或整合進工具列。確保 Blockly 觸控支援在全寬下正常運作。」

## 釐清事項

### 會議 2026-03-11

- Q: Variables 面板在行動裝置上的位置？ → A: 放在主控台分頁內，以子分頁（sub-tab）切換 Console 和 Variables。
- Q: 漢堡選單的互動模式？ → A: 下拉式覆蓋（dropdown overlay），從工具列向下展開，點擊外部關閉。

## 使用者情境與測試 *(必填)*

### 使用者故事 1 — 行動裝置上全寬積木編輯（優先級：P1）

學生在手機上打開 Semorphe 練習 C++ 積木程式設計。不再是擁擠的分割畫面，而是看到一個全寬的 Blockly 工作區，可以舒適地瀏覽工具箱、拖曳積木、連接積木、雙指縮放。積木面板填滿整個螢幕，方便用觸控手勢選取和排列積木。

**優先級理由**: Semorphe 的核心價值是積木式程式設計。如果積木在行動裝置上無法使用，整個行動裝置體驗就失敗了。這是其他所有故事的基礎。

**獨立測試**: 在手機寬度的瀏覽器（≤768px）上打開 Semorphe。Blockly 工作區填滿螢幕。使用者可以打開工具箱、拖曳積木、連接積木、雙指縮放，沒有任何佈局問題。

**驗收情境**:

1. **假設**使用者在螢幕寬度 ≤768px 的裝置上打開 Semorphe，**當**頁面載入完成，**則** Blockly 工作區佔滿整個螢幕寬度（減去最小框架），且分割面板分隔線不可見。
2. **假設**使用者在行動裝置上檢視積木分頁，**當**他們點擊 Blockly 工具箱，**則**工具箱開啟，分類和積木大到足以準確點擊（最小 44px 觸控目標）。
3. **假設**使用者在行動裝置上編輯積木，**當**他們雙指縮放或平移工作區，**則**手勢順暢運作，不會與頁面捲動或 tab 切換衝突。

---

### 使用者故事 2 — 面板之間的 Tab 切換（優先級：P1）

學生想查看積木產生的程式碼。他們點擊螢幕底部的「程式碼」分頁，切換到程式碼編輯器，同樣填滿整個螢幕。接著可以點「積木」返回，或點「主控台」查看輸出。主控台分頁內含 Console 和 Variables 兩個子分頁，可切換查看輸出或變數狀態。每個分頁上的同步指示器顯示內容是否有更新。

**優先級理由**: 沒有 tab 切換，使用者就被困在單一面板，無法存取其他視圖。這與故事 1 同等重要——兩者共同構成最小可行的行動裝置體驗。

**獨立測試**: 在行動裝置寬度下，點擊 3 個分頁（積木/程式碼/主控台），驗證每個面板都以全寬顯示。在主控台分頁中切換 Console 和 Variables 子分頁。驗證當非活躍面板的內容更新時，分頁上出現同步指示。

**驗收情境**:

1. **假設**使用者在任一分頁上，**當**他們點擊底部 tab bar 的另一個分頁，**則**新面板以全寬順暢取代目前面板。
2. **假設**使用者編輯積木後切換到程式碼分頁，**當**程式碼面板出現，**則**產生的程式碼反映積木的變更（自動同步），並顯示任何待處理的同步狀態。
3. **假設**自動同步在使用者位於積木分頁時更新了程式碼，**當**他們查看底部 tab bar，**則**程式碼分頁顯示一個小指示器（圓點或徽章），表示內容已更新。
4. **假設**使用者在主控台分頁檢視輸出，**當**他們點擊「積木」，**則**積木工作區恢復到先前的狀態（捲動位置、縮放比例）。
5. **假設**使用者在主控台分頁上，**當**他們切換 Console/Variables 子分頁，**則**對應的面板內容顯示，且子分頁狀態在離開主控台分頁後仍保留。

---

### 使用者故事 3 — 行動裝置上的簡化工具列（優先級：P2）

在行動裝置上，工具列只直接顯示最重要的控制項（執行按鈕、同步按鈕）。次要控制項（語言選擇器、程式碼風格選擇器、積木風格選擇器、Topic 選擇器、語言設定選擇器）收納在下拉式覆蓋選單中（從工具列向下展開，點擊外部關閉）。這避免了工具列溢出，保持介面簡潔。

**優先級理由**: 在小螢幕上溢出的雜亂工具列會降低可用性，但不會阻擋核心功能。使用者即使工具列不理想，仍可使用積木和程式碼，因此這是重要但非關鍵的。

**獨立測試**: 在行動裝置寬度下打開 Semorphe。驗證只有執行和同步按鈕可見。點擊選單按鈕，驗證下拉式覆蓋選單展開，所有次要控制項都可存取。點擊選單外部，驗證選單關閉。

**驗收情境**:

1. **假設**使用者在行動裝置上，**當**頁面載入完成，**則**工具列只顯示應用程式標題、執行按鈕和同步按鈕，以及一個漢堡/選單圖示供額外選項。
2. **假設**使用者點擊漢堡選單，**當**選單開啟，**則**下拉式覆蓋選單從工具列向下展開，列出所有次要控制項（語言、程式碼風格、積木風格、Topic、語言設定）且可使用。
3. **假設**使用者從選單選擇不同的程式碼風格，**當**選擇完成，**則**風格變更立即套用，選單關閉。
4. **假設**選單已開啟，**當**使用者點擊選單外部區域，**則**選單關閉。

---

### 使用者故事 4 — 桌面佈局維持不變（優先級：P2）

桌面使用者（螢幕寬度 >768px）看到與之前完全相同的佈局——左右分割面板含可拖曳分隔線、完整工具列、狀態列。桌面體驗沒有退化。

**優先級理由**: 保護現有桌面體驗至關重要，但由於不需要新開發（只需確保無退化），這是一個 P2 守護條件。

**獨立測試**: 在桌面瀏覽器（≥769px 寬度）上打開 Semorphe。驗證分割面板、分隔線、工具列和狀態列都出現且功能與目前版本完全一致。

**驗收情境**:

1. **假設**使用者在螢幕寬度 ≥769px 的裝置上打開 Semorphe，**當**頁面載入完成，**則**佈局顯示分割面板，左邊 Blockly、右邊 Monaco、可拖曳分隔線、完整工具列和狀態列。
2. **假設**使用者將桌面瀏覽器視窗從寬縮到窄，**當**寬度低於 769px，**則**佈局順暢轉換為 tab 切換模式，不需重新載入頁面。
3. **假設**使用者將瀏覽器視窗從窄拉到寬，**當**寬度超過 768px，**則**佈局順暢轉換回分割面板模式，各面板保留其內容和狀態。

---

### 使用者故事 5 — 行動裝置上的狀態列（優先級：P3）

在行動裝置上，桌面風格的狀態列（語言、風格、積木風格、Topic、語言設定）被隱藏以節省垂直空間。這些資訊可透過漢堡選單的下拉式覆蓋存取。

**優先級理由**: 狀態列僅供資訊參考，其內容已可透過工具列選單存取。在行動裝置上隱藏它是潤飾項目。

**獨立測試**: 在行動裝置寬度下打開 Semorphe。驗證狀態列不可見。驗證相同資訊可透過選單存取。

**驗收情境**:

1. **假設**使用者在行動裝置上，**當**頁面載入完成，**則**狀態列不可見，為活躍面板提供更多垂直空間。
2. **假設**使用者想在行動裝置上查看目前的語言/風格設定，**當**他們打開漢堡選單，**則**目前設定會顯示在下拉式覆蓋選單中。

---

### 邊界案例

- 使用者將裝置從直向旋轉到橫向（或反之）時會怎樣？佈局應自適應：如果橫向寬度超過 768px，切換到分割面板模式；否則留在 tab 模式。
- 使用者正在拖曳積木時，瀏覽器跨越斷點 resize 會怎樣？積木操作應正常完成，佈局轉換應在互動結束後發生。
- 使用者有未儲存/未同步的變更時切換分頁會怎樣？變更應保留——tab 切換不應觸發資料遺失。
- 底部 tab bar 與 Blockly 工具箱在螢幕底部重疊時會怎樣？Blockly 工作區面積必須考慮 tab bar 的高度，積木不應被隱藏在 tab bar 後面。
- 在折疊式裝置或平板的分割螢幕模式下會怎樣？佈局應根據實際可用寬度，而非裝置類型來判斷。

## 需求 *(必填)*

### 功能需求

- **FR-001**: 當視窗寬度 ≤768px 時，系統必須顯示含 3 個分頁（積木、程式碼、主控台）的底部 tab bar。
- **FR-002**: 在行動裝置上，系統必須一次只顯示一個面板，佔滿全部可用寬度和高度（減去工具列和 tab bar）。
- **FR-003**: 當視窗寬度 ≥769px 時，系統必須保留目前的桌面分割面板佈局。
- **FR-004**: 當視窗跨越 768px 斷點 resize 時，系統必須在行動和桌面佈局之間轉換，不需重新載入頁面。
- **FR-005**: 在行動裝置上，系統必須只在工具列中顯示主要操作（執行、同步）和一個選單按鈕。
- **FR-006**: 在行動裝置上，系統必須提供下拉式覆蓋選單（從工具列向下展開、點擊外部關閉），包含所有次要工具列控制項。
- **FR-007**: 在行動裝置視窗上，系統必須隱藏狀態列。
- **FR-008**: 當非活躍分頁的內容已更新時，系統必須在該分頁上顯示同步指示器（徽章/圓點）。
- **FR-009**: 切換分頁時，系統必須保留面板狀態（捲動位置、縮放比例、游標位置）。
- **FR-010**: 系統必須確保 Blockly 工作區不與底部 tab bar 重疊（工作區高度需扣除 tab bar）。
- **FR-011**: Tab bar 觸控目標高度必須至少 44px，符合無障礙設計。
- **FR-012**: 系統必須支援 Blockly 工作區中的標準觸控手勢（拖曳、雙指縮放、平移），不與 tab bar 或頁面手勢衝突。
- **FR-013**: 主控台分頁內必須包含 Console 和 Variables 兩個子分頁，可切換顯示。

## 成功標準 *(必填)*

### 可衡量成果

- **SC-001**: 使用者能在 375px 寬的視窗（iPhone SE 尺寸）上成功拖曳和連接積木，不會出現誤觸或手勢衝突。
- **SC-002**: 使用者能在 3 個分頁之間切換，每次切換在 1 秒內完成並顯示全寬內容。
- **SC-003**: 所有現有桌面佈局測試持續通過，無退化。
- **SC-004**: Blockly 工具箱在行動裝置上完全可用——所有分類可開啟，積木可單擊選取。
- **SC-005**: 佈局在瀏覽器視窗跨越 768px 斷點 resize 時，雙向正確適應。
- **SC-006**: 在任何行動裝置視窗尺寸（320px 至 768px）上，沒有內容被隱藏在 tab bar 或工具列後方。

## 假設

- 768px 斷點適合區分行動裝置和桌面。這符合常見的平板直向/手機橫向寬度。
- Blockly 內建的觸控支援（拖曳、雙指縮放）已足夠，不需自訂手勢處理，只需正確的容器大小。
- Monaco 編輯器在行動裝置全寬下可以堪用（唯讀或有限編輯可接受——完整的行動裝置程式碼編輯不是主要使用案例）。
- 底部 tab bar 的方式（類似原生行動應用程式）對目標受眾（學生）比滑動手勢更直覺。
