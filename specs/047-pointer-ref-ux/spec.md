# Feature Specification: C++ 指標與參照積木 UX 重設計

**Feature Branch**: `047-pointer-ref-ux`
**Created**: 2026-03-16
**Status**: Draft
**Input**: Redesign C++ pointer and reference block UX in Semorphe for beginner-friendly experience

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 指標宣告時可選擇初始化 (Priority: P1)

學習者想宣告一個指標變數並同時初始化指向某個變數，就像寫 `int* ptr = &x;` 一樣自然。目前的積木只有空指標宣告（TYPE + NAME），沒有初始化槽位，學生必須額外拖一個指標賦值積木才能完成初始化，這不符合真實 C++ 的使用習慣。

**Why this priority**: 指標宣告是學習指標的第一步，如果第一步就不直覺，後續所有指標操作都會讓學生困惑。

**Independent Test**: 學生可以單獨使用「宣告指標」積木完成 `int* ptr = &x;` 的效果，不需要其他積木配合。

**Acceptance Scenarios**:

1. **Given** 學生拖出「宣告指標」積木, **When** 不連接任何初始值, **Then** 產生 `int* ptr;`（空指標宣告）
2. **Given** 學生拖出「宣告指標」積木, **When** 在初始化槽位連接一個「取得位址」積木, **Then** 產生 `int* ptr = &x;`
3. **Given** 學生拖出「宣告指標」積木, **When** 在初始化槽位連接 `nullptr`, **Then** 產生 `int* ptr = nullptr;`

---

### User Story 2 - 參照宣告使用「別名」語義 (Priority: P1)

學習者想建立一個參照變數作為另一個變數的別名，就像寫 `int& ref = x;` 一樣。目前積木使用原始的 `&` 符號，學生不理解 `&` 在宣告中代表參照而非取址。積木應該用描述性標籤（如「建立別名」或「參照」），讓學生理解參照的語義是「給變數取另一個名字」。

**Why this priority**: 參照與指標並列為 C++ 記憶體概念的基礎，且參照的 `&` 符號與取址的 `&` 容易混淆，是初學者最大的困惑來源之一。

**Independent Test**: 學生可以獨立使用「建立參照」積木完成 `int& ref = x;`，且從積木外觀就能理解「這是給變數取別名」。

**Acceptance Scenarios**:

1. **Given** 學生拖出「建立參照」積木, **When** 連接目標變數, **Then** 產生 `int& ref = x;`
2. **Given** 學生看到「建立參照」積木, **When** 閱讀積木標籤和 tooltip, **Then** 能理解參照是變數的別名，而非取得位址

---

### User Story 3 - 解參照與取址操作有描述性標籤 (Priority: P2)

學習者想讀取指標指向的值（`*ptr`）或取得變數的記憶體位址（`&var`）。目前積木存在但標籤使用原始符號 `*` 和 `&`，學生不知道何時該用哪個。積木應使用描述性標籤（如「取得 %1 指向的值」、「取得 %1 的位址」），並提供 tooltip 解釋。

**Why this priority**: 這兩個操作是指標日常使用的核心，但符號的多義性（`*` 既是乘法又是解參照，`&` 既是位元 AND 又是取址又是參照宣告）是學生最大的障礙。

**Independent Test**: 學生在看到積木時，不需要事先知道 `*` 和 `&` 的意義，就能從標籤理解操作的效果。

**Acceptance Scenarios**:

1. **Given** 學生拖出「取得指向的值」積木, **When** 連接一個指標變數, **Then** 產生 `*ptr` 並在 tooltip 說明「解參照：讀取指標 ptr 所指向的記憶體內容」
2. **Given** 學生拖出「取得位址」積木, **When** 連接一個一般變數, **Then** 產生 `&x` 並在 tooltip 說明「取址：取得變數 x 在記憶體中的位址」

---

### User Story 4 - 指標與參照的視覺區分 (Priority: P2)

學習者在工具箱中瀏覽時，應能一眼區分指標相關積木和參照相關積木。目前兩者都在同一個分類中且外觀類似，容易混淆。應透過子分類標籤、顏色深淺或圖示來區分。

**Why this priority**: 視覺區分是降低認知負載的基本手段。當指標和參照積木混在一起且外觀相似時，學生必須逐一閱讀標籤才能找到需要的積木。

**Independent Test**: 給 5 個未接觸過 C++ 的學生看工具箱截圖，80% 以上能正確指出哪些是指標積木、哪些是參照積木。

**Acceptance Scenarios**:

1. **Given** 學生打開「指標與參照」工具箱分類, **When** 瀏覽積木列表, **Then** 指標積木和參照積木有明確的視覺區分（子分類標籤或色彩差異）
2. **Given** 學生搜尋「指標」, **When** 查看搜尋結果, **Then** 只顯示指標相關積木，不混入參照積木

---

### User Story 5 - 指標重新賦值語義清晰 (Priority: P3)

學習者想將指標重新指向另一個變數（`ptr = &y;`）。目前的「指標賦值」積木標籤應清楚表達「讓指標指向另一個目標」，而非看起來像是「修改指標指向的值」（那是 `*ptr = value;`）。

**Why this priority**: 「改變指標的指向」和「改變指標指向的值」是指標學習中最容易混淆的兩個操作，但只要前面的 US1-US3 設計好，此問題的嚴重度會降低。

**Independent Test**: 學生能正確選擇「重新指向」積木（而非「透過指標賦值」積木）來完成 `ptr = &y;`。

**Acceptance Scenarios**:

1. **Given** 學生想讓 ptr 指向 y, **When** 瀏覽工具箱, **Then** 能找到「讓 %1 指向 %2」積木並正確使用
2. **Given** 學生混淆「重新指向」和「透過指標修改值」, **When** 閱讀兩個積木的 tooltip, **Then** 能理解差異

---

### Edge Cases

- 指標宣告的初始化槽位留空時，tooltip 顯示警告提示「未初始化的指標可能導致程式崩潰」（積木外觀不變色）
- `void*` 指標的宣告和使用（型別系統更複雜）
- 指標的指標（`int** pp`）是否需要支援？
- 函式指標是否在此範圍內？
- `const` 修飾與指標結合（`const int*` vs `int* const`）的 UX

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 指標宣告積木 MUST 提供可選的初始化輸入槽位（input_value），允許學生在宣告時連接初始值表達式
- **FR-002**: 參照宣告積木 MUST 使用描述性中文標籤（如「建立 %1 型別的參照 %2 指向 %3」），不顯示原始的 `&` 符號
- **FR-003**: 解參照積木 MUST 使用描述性標籤並附帶原始符號（如「取得 %1 指向的值 (*)」），讓學生同時學習語義與語法
- **FR-004**: 取址積木 MUST 使用描述性標籤並附帶原始符號（如「取得 %1 的位址 (&)」），讓學生同時學習語義與語法
- **FR-005**: 所有指標/參照積木的 tooltip MUST 提供額外說明，包含類比或比喻幫助理解（如「指標像是寫著地址的便條紙」）
- **FR-006**: 指標相關積木和參照相關積木 MUST 有視覺區分（透過子分類標籤、色彩變體或前綴圖示）
- **FR-007**: 指標賦值（`ptr = &y;`）和透過指標修改值（`*ptr = val;`）MUST 是兩個不同的積木，標籤清楚區分語義
- **FR-008**: 所有新增/修改的積木 MUST 有完整的 i18n 支援（zh-TW 和 en），標籤遵循 Semorphe i18n 風格規範（描述式、無原始語法符號）
- **FR-009**: 所有修改的積木 MUST 通過完整的 lift → generate → round-trip 驗證，確保雙向轉換正確
- **FR-010**: 積木變更 MUST 向後相容——已存在的使用舊積木的學生專案在載入時能自動遷移到新積木結構

### Key Entities

- **指標變數（Pointer Variable）**: 儲存記憶體位址的變數，有型別（指向什麼型別）、名稱、可選的初始值
- **參照變數（Reference Variable）**: 既有變數的別名，有型別、名稱、必須的目標變數
- **解參照操作（Dereference）**: 讀取指標指向的記憶體內容，輸入為指標表達式，輸出為值
- **取址操作（Address-of）**: 取得變數的記憶體位址，輸入為變數，輸出為指標值

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 80% 的初學者（未學過指標）能在 3 分鐘內正確使用積木完成 `int* ptr = &x; cout << *ptr;` 的等效操作
- **SC-002**: 所有指標/參照相關積木的 i18n 標籤 100% 符合 Semorphe 風格規範（無原始語法符號、描述式標籤）
- **SC-003**: 指標/參照概念的 round-trip 測試通過率達 100%（lift → generate → re-lift 語義樹結構等價）
- **SC-004**: 工具箱中指標積木和參照積木的視覺區分讓 80% 的使用者能正確分類（無需閱讀標籤細節）
- **SC-005**: 舊版專案載入後，所有既有的指標/參照積木自動遷移成功率 100%，無資料遺失

## Clarifications

### Session 2026-03-16

- Q: 積木標籤是否應保留 C++ 原始符號作為輔助提示？ → A: 描述性標籤 + 括號內附原始符號（如「取得 %1 指向的值 (*)」）
- Q: 指標宣告初始化槽位留空時是否顯示警告？ → A: tooltip 警告（hover 時提示「未初始化的指標可能導致程式崩潰」，積木外觀不變）

## Assumptions

- 目標使用者為 C++ 初學者（高中到大學程式設計課程學生）
- 函式指標和指標的指標（`int**`）不在本次範圍內，可作為後續擴展
- `const` 與指標結合的 UX（`const int*` vs `int* const`）已由現有的 `const` 積木處理，不在本次範圍內
- void 指標（`void*`）暫不處理，屬進階議題
- 向後相容的遷移邏輯會在積木載入時自動執行（利用 Blockly 的 mutation/extraState 機制）

## Scope Boundaries

**包含**:
- 指標宣告積木的初始化槽位新增
- 參照宣告積木的標籤重設計
- 解參照和取址積木的標籤重設計
- tooltip 增強（類比和說明）
- i18n 翻譯（zh-TW、en）
- 視覺區分（子分類或色彩）
- lift/generate/round-trip 驗證
- 向後相容遷移

**不包含**:
- 函式指標
- 指標的指標（`int**`）
- `void*` 指標
- 指標算術（`ptr + 1`、`ptr++`）
- 智慧指標（`unique_ptr`、`shared_ptr`）
- `const` 修飾與指標的組合 UX
