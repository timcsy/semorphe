# Feature Specification: 概念式積木系統重新設計

**Feature Branch**: `002-concept-blocks-redesign`
**Created**: 2026-03-03
**Status**: Draft
**Input**: 依照第一性原理重新設計積木系統，使積木表達「概念」而非「語法」，讓學生透過積木理解程式碼的結構與意義，而非重複閱讀語法。

## Clarifications

### Session 2026-03-03

- Q: 002 相對於 001 是完全取代、漸進重構、還是獨立並行？ → A: 完全取代——002 取代 001 的積木定義和轉換層，不保留舊格式相容性
- Q: 運算符號（+, >=, == 等）在積木上是否允許顯示？ → A: 數學通用符號（算術、比較）允許直接顯示；僅禁止程式專用符號（<< >> && ; {} 等）
- Q: 共用積木的完整清單？ → A: 不設上限，以下 21 塊——建立變數、設定變數值、變數參考、數字常數、字串常數、算術運算、比較運算、邏輯運算、如果就、如果就否則、計數式重複、條件式重複、中斷、繼續、定義函式、呼叫函式、回傳值、輸出、讀取輸入、建立陣列、陣列存取
- Q: 不符合計數模式的 for 迴圈如何處理？ → A: 降級為語言特殊的「三段式 for」積木（顯示 init/條件/更新 三個欄位，仍用自然語言標籤）
- Q: 積木與程式碼的對照高亮互動範圍？ → A: 本次實作雙向——點積木高亮對應程式碼行 + 點程式碼高亮對應積木

### 設計原則

- **第一性原理**：程式的本質是三件事——記住東西（資料）、做決定（控制流程）、重複做事（迴圈與函式）。積木應圍繞這三個本質概念設計。
- **共用 vs 特殊的判斷標準**：學生腦中建立的心智模型是否相同。若相同（例如「重複做某件事」），則共用積木；若不同（例如 C 的指標 vs Python 沒有指標），則語言特殊積木。
- **積木是翻譯官**：積木負責用自然語言呈現「概念」，右側 CodeMirror 負責呈現真實程式碼。兩者對照，讓學生同時建立「理解」與「認識語法」。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 學生透過概念積木理解程式碼結構 (Priority: P1)

一位 APCS 考生在右側程式碼編輯器中貼上一段 C++ 程式碼。系統解析後，左側 Blockly 編輯器顯示以自然語言（繁體中文）描述的概念積木。學生可以透過積木的顏色、嵌套結構、自然語言標籤來理解程式碼的邏輯結構（例如看到「重複：i 從 0 到 9」就理解這是一個計數迴圈），同時對照右側的 `for (int i = 0; i < 10; i++)` 來認識語法。

**Why this priority**: 這是整個系統的核心價值——讓學生「看懂程式碼」。沒有概念式積木，系統就只是另一個語法展示工具。

**Independent Test**: 可以貼上一段典型 APCS 程式碼（含變數、迴圈、條件判斷、輸入輸出），驗證積木顯示的是概念描述而非程式語法。

**Acceptance Scenarios**:

1. **Given** 學生貼上含有 `for (int i = 0; i < 10; i++)` 的 C++ 程式碼，**When** 系統完成解析，**Then** Blockly 端顯示以自然語言描述的迴圈積木（例如「重複：i 從 0 到 9」），而非語法格式的 `for ( ; ; )`
2. **Given** 學生貼上含有 `cin >> x;` 的程式碼，**When** 系統完成解析，**Then** 積木顯示為概念描述（例如「讀取輸入 → x」），而非 `cin >> x`
3. **Given** 學生貼上含有 `if (score >= 60)` 的程式碼，**When** 系統完成解析，**Then** 積木顯示為概念描述（例如「如果 score >= 60 就...」），且條件分支的嵌套結構一目瞭然
4. **Given** 積木與程式碼同時顯示在畫面上，**When** 學生點選某塊積木，**Then** 右側程式碼編輯器自動高亮對應的程式碼行
5. **Given** 積木與程式碼同時顯示在畫面上，**When** 學生在程式碼編輯器中點選某行程式碼，**Then** 左側 Blockly 編輯器自動高亮對應的積木

---

### User Story 2 - 學生使用共用積木拖拉組合程式邏輯 (Priority: P2)

學生從 Blockly 工具箱中拖出概念積木（如「建立變數」、「如果...就...」、「重複」、「輸出」等），用嵌套和連接的方式組合程式邏輯。系統即時在右側產生對應的 C++ 程式碼。學生不需要記住任何語法，只需要用自然語言理解每塊積木的意義。

**Why this priority**: 拖拉組合是 Blockly 的原生強項，也是學生從「理解概念」到「動手做」的下一步。

**Independent Test**: 可以從工具箱拖出積木組合一個完整的小程式（如印出 1 到 10），驗證產出的 C++ 程式碼語法正確。

**Acceptance Scenarios**:

1. **Given** 工具箱中有所有共用概念積木，**When** 學生瀏覽工具箱，**Then** 每塊積木都以自然語言標示其概念（如「建立 int 變數 x = 0」），且按概念類別（資料、流程控制、函式、輸入輸出、運算、陣列）分類
2. **Given** 學生拖拉積木組合了一個完整程式，**When** 組合完成，**Then** 右側即時產生語法正確的 C++ 程式碼
3. **Given** 學生修改了積木上的欄位值（如將變數名稱從 x 改為 score），**When** 修改完成，**Then** 右側程式碼即時更新

---

### User Story 3 - 共用積木支援多語言程式碼生成 (Priority: P3)

同一套共用概念積木可以產生不同程式語言的程式碼。學生用相同的積木組合邏輯，系統根據選定的目標語言產生對應的程式碼。這讓學生理解「程式邏輯跨語言相通」，只是語法不同。

**Why this priority**: 多語言支援是概念式積木設計的核心優勢，但依賴共用積木先完成。目前優先支援 C/C++，未來擴充 Python、Java。

**Independent Test**: 可以用同一組積木，切換目標語言後，驗證產出不同語言的程式碼。初期僅需驗證 C/C++ 的完整支援，以及架構上預留多語言擴充能力。

**Acceptance Scenarios**:

1. **Given** 學生用共用積木組合了一段程式邏輯，**When** 目標語言設定為 C++，**Then** 右側產生正確的 C++ 程式碼
2. **Given** 積木定義與語言無關，**When** 新增一個語言的 Parser 和 Generator，**Then** 不需要修改任何積木定義即可支援新語言的雙向轉換
3. **Given** 某個概念在目標語言中不存在（例如 Python 無指標），**When** 學生切換到該語言，**Then** 工具箱中不顯示該語言不支援的語言特殊積木

---

### User Story 4 - 語言特殊積木處理獨有概念 (Priority: P4)

對於特定語言獨有的概念（如 C/C++ 的指標、格式化字串 printf/scanf），系統提供語言特殊積木。這些積木僅在對應語言模式下出現於工具箱，且積木標籤描述的是該語言特有的心智模型（如「取得 x 的記憶體位址」而非 `&x`）。

**Why this priority**: 語言特殊積木補充共用積木無法涵蓋的邊角，但它們只影響少數場景，非核心路徑。

**Independent Test**: 可以在 C++ 模式下驗證語言特殊積木出現在工具箱中，且能正確生成對應程式碼。

**Acceptance Scenarios**:

1. **Given** 系統處於 C++ 語言模式，**When** 學生瀏覽工具箱，**Then** 工具箱中顯示 C++ 特殊積木（如指標、printf/scanf、struct、namespace、do-while、switch）
2. **Given** 學生使用 C++ 的「格式化輸出」積木，**When** 組合完成，**Then** 產生正確的 `printf(...)` 程式碼
3. **Given** 程式碼中包含 C++ 特有語法（如指標操作），**When** 系統解析該程式碼，**Then** 以語言特殊積木呈現，積木標籤描述概念（如「取得 x 的記憶體位址」）

---

### Edge Cases

- 當程式碼混用共用概念和語言特有語法時（例如 for 迴圈內有指標操作），系統正確地將共用部分以共用積木呈現，語言特有部分以特殊積木呈現
- 當 for 迴圈不符合計數模式（例如 `for (int i = 0; i < n; i += 2)` 或 `for (;;)`）時，系統降級為語言特殊的「三段式 for」積木（以自然語言標籤顯示 初始化/條件/更新 三個欄位），不使用原始碼積木
- 當共用積木的欄位值（如型別）在目標語言中不適用時（例如 Python 不需要型別宣告），該欄位自動隱藏或設為可選
- 當程式碼包含無法映射到任何共用或特殊積木的語法時，系統以「原始碼積木」保留該最小片段，不遺失內容

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 提供一套與程式語言無關的共用積木定義，完整清單如下：
  - 資料：建立變數（含型別與初始值）、設定變數值、變數參考、數字常數、字串常數
  - 流程控制：如果...就...、如果...就.../否則...、計數式重複（i 從 A 到 B）、條件式重複（當...持續執行）、中斷（跳出迴圈）、繼續（跳至下一次）
  - 函式：定義函式、呼叫函式、回傳值
  - 輸入輸出：輸出、讀取輸入
  - 運算：算術運算（+ - * / %）、比較運算（> < >= <= == !=）、邏輯運算（而且/或者/不是）
  - 陣列：建立陣列、陣列存取
- **FR-002**: 共用積木 MUST 以自然語言（繁體中文）描述概念。數學通用符號（`+` `-` `*` `/` `%` `>` `<` `>=` `<=` `==` `!=`）允許直接顯示；程式專用符號（如 `;` `{}` `<<` `>>` `&&` `||` `&` `*`指標 等）不得出現，須以自然語言替代（如「而且」「或者」「不是」）
- **FR-003**: 系統 MUST 支援語言特殊積木，這些積木僅在對應的語言模式下出現於工具箱。初期 MUST 支援 C/C++ 特殊積木：三段式 for 迴圈（處理非計數模式）、指標操作、printf/scanf 格式化輸入輸出、struct、namespace、do-while、switch（含 fall-through 語意）
- **FR-004**: 積木定義 MUST 與特定程式語言解耦——積木定義不包含任何語言特定的程式碼模板，程式碼生成邏輯完全由各語言的 Generator 負責
- **FR-005**: 每個支援的語言 MUST 各自擁有獨立的 Parser（程式碼 → 積木）和 Generator（積木 → 程式碼），且不影響其他語言的實作
- **FR-006**: 系統 MUST 保持與 001 版本相同的雙向即時同步能力——積木端修改即時反映到程式碼端，程式碼端修改即時反映到積木端
- **FR-007**: 共用積木的工具箱 MUST 按概念類別分類：資料、流程控制、函式、輸入輸出、運算、陣列
- **FR-008**: 系統 MUST 支援語言模式切換，切換後工具箱自動更新（顯示/隱藏語言特殊積木），且共用積木的 Generator 自動切換到對應語言
- **FR-009**: 對於計數式 for 迴圈，共用積木 MUST 將「初始值」「結束值」「步進」抽象為獨立欄位，而非暴露 init/condition/update 三段式語法
- **FR-010**: 對於輸入輸出，共用積木 MUST 將概念抽象為「輸出某個值」和「讀取輸入到某個變數」，隱藏各語言的具體 I/O 語法（如 cin/cout、scanf/printf、print/input）
- **FR-011**: 系統 MUST 繼續支援使用者上傳自訂積木定義檔來擴充積木集合
- **FR-012**: 當程式碼包含無法映射的語法結構時，系統 MUST 以「原始碼積木」保留該最小片段，不遺失內容
- **FR-013**: 共用積木中的「變數宣告」積木 MUST 支援型別欄位可選化——在需要型別宣告的語言（C/C++、Java）中必填，在不需要的語言（Python）中自動隱藏
- **FR-014**: 系統 MUST 支援雙向對照高亮——點選積木時高亮對應的程式碼行，點選程式碼行時高亮對應的積木

### Key Entities

- **共用積木定義（Universal Block Definition）**: 與語言無關的積木規格，包含積木外觀（自然語言標籤、顏色、欄位定義、連接類型）和概念類別。不包含程式碼模板，作為系統的中間表示（IR）
- **語言特殊積木定義（Language-Specific Block Definition）**: 綁定特定語言的積木規格，除了外觀定義外，還標記其所屬語言。僅在對應語言模式下可見
- **語言模組（Language Module）**: 一個語言的完整支援單元，包含 Parser（程式碼 → 積木）、Generator（積木 → 程式碼）、以及該語言的特殊積木定義集合
- **概念類別（Concept Category）**: 積木的分類維度，基於計算的三個本質——資料、控制流程、重複/函式——加上輸入輸出、運算、陣列等衍生類別

## Assumptions

- 初期僅完整支援 C/C++ 語言模組，Python 和 Java 的支援作為未來擴充，但架構上必須預留
- 積木的自然語言標籤以繁體中文為主，未來可考慮國際化但不在本次範圍內
- 計數式 for 迴圈的抽象化（「i 從 A 到 B」）能涵蓋 APCS 常見的 80% 以上迴圈場景；不常見的複雜迴圈模式可降級為語言特殊積木或原始碼積木
- 002 完全取代 001 的積木定義和轉換層，不需要與 001 舊格式相容。Web UI 架構（左右分割、CodeMirror、Blockly）維持不變，重構聚焦在積木定義層和轉換層

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 學生看到概念積木後能正確描述程式邏輯，而不需要閱讀右側程式碼。可透過使用者測試驗證：展示積木組合，請學生口述程式行為，正確率達 80% 以上
- **SC-002**: 共用積木 MUST 涵蓋 APCS 實作題常見語法結構的 80% 以上，僅少數語言特有語法需要特殊積木
- **SC-003**: 新增一個語言模組（Parser + Generator）不需要修改任何共用積木定義，僅需撰寫新的語言模組
- **SC-004**: 在 C++ 模式下，Block → Code → Block 的往返轉換 MUST 保持語意等價
- **SC-005**: 積木端修改後程式碼端在 1 秒內更新；程式碼端修改後積木端在 2 秒內更新（與 001 版本相同）
- **SC-006**: 共用積木 MUST 涵蓋 FR-001 定義的 21 塊完整清單；語言特殊積木數量不設硬性上限，但工具箱分類須清晰，避免認知負擔
